

## 一、 基础规范（Design Tokens）

在进入组件之前，必须定义好最基础 relocation 的视觉粒子。这些规范决定了组件的皮肤和骨架。

### 1. 颜色系统 (Color)

* **品牌主色 (Primary):** 用于核心操作、高亮状态、关键按钮。
* **功能色 (Functional):** * 成功 (Success): 绿色，用于表单校验通过、成功提示。
* 警告 (Warning): 黄色/橙色，用于风险提示。
* 错误 (Error): 红色，用于报错、删除等危险操作。


* **中性色 (Neutral):** 用于文字、背景、边框、分割线（通常是一组不同饱和度的灰色）。

### 2. 字体系统 (Typography)

* **字阶与行高:** 建立统一的字号梯度（如 12px, 14px, 16px, 20px, 24px, 32px），并严格对应行高（通常为字号的 1.2 ~ 1.5 倍），确保文本可读性。
* **字重 (Font Weight):** 通常分为 Regular（常规）、Medium（中黑）、Bold（粗体）。

### 3. 栅格与间距 (Grid & Spacing)

* **8px 原则:** 所有组件的间距、内边距（Padding）、外边距（Margin）及高度，建议遵循 8px 的倍数（如 8, 16, 24, 32, 40），特殊小间距可用 4px。

## 二、 常用 UI 组件列表

根据组件的功能属性，通常将其分为以下六大类：

### 1. 通用组件 (General)

* **按钮 (Button):** 触发特定交互。分为主按钮（Primary）、次按钮（Secondary）、虚线框按钮（Dashed）、文本按钮（Text）和链接按钮（Link）。
* **图标 (Icon):** 语义化的矢量图形，辅助文字表达。

### 2. 布局组件 (Layout)

* **栅格 (Grid):** 24 栅格系统，用于实现响应式布局。
* **布局容器 (Layout):** 协助搭建页面的整体骨架（如 Header, Sider, Content, Footer）。
* **间距 (Space):** 统一设置组件之间的间距。

### 3. 导航组件 (Navigation)

* **面包屑 (Breadcrumb):** 显示当前页面在系统层级中的位置，并能返回任一上级。
* **导航菜单 (Menu):** 为网站提供通行的导航结构。
* **标签页 (Tabs):** 在同一区域内切换不同子内容。
* **分页器 (Pagination):** 用于大数据量时分批加载。

### 4. 数据录入组件 (Data Entry)

* **输入框 (Input) / 文本域 (Textarea):** 基础的文本输入工具。
* **单选框 (Radio) / 多选框 (Checkbox):** 用于在一组备选项中进行单选或多选。
* **选择器 (Select):** 下拉选择菜单，当选项较多时代替 Radio。
* **开关 (Switch):** 两种互斥状态的切换（开/关）。
* **日期/时间选择器 (DatePicker):** 选择特定日期或时间段。
* **表单 (Form):** 由上述多个录入组件组合而成，用于数据收集和校验。

### 5. 数据展示组件 (Data Display)

* **表格 (Table):** 用于展示结构化数据，支持排序、筛选、分页。
* **折叠面板 (Collapse):** 折叠隐藏大段内容，保持页面整洁。
* **卡片 (Card):** 将信息聚合在独立的容器中。
* **化身 (Avatar):** 用图标、图片或字符代表用户/实体。

### 6. 反馈组件 (Feedback)

* **对话框 (Modal / Dialog):** 遮罩弹窗，需要用户强制交互。
* **抽屉 (Drawer):** 从屏幕边缘滑出的浮层，不中断主流程的情况下展示更多详情。
* **全局提示 (Message):** 页面顶部居中的轻量级信息反馈，自动消失。
* **通知提醒框 (Notification):** 边缘弹出的通知，通常含较多系统级信息。

## 三、 UI 组件设计规范（交互与视觉规范）

为了保证组件在落地时不走样，设计时必须遵循以下规范规则：

### 1. 状态规范 (Component States)

每个交互组件（特别是按钮和表单项）都必须设计并定义以下 **5 种标准状态** 的视觉表现：

* **Normal（常态）:** 默认未激活状态。
* **Hover（悬停）:** 鼠标指针悬停在组件上时的反馈（通常颜色加深或变浅 10%）。
* **Active（激活/点击）:** 鼠标按下瞬间的反馈。
* **Focus（聚焦）:** 键盘导航或输入框正在输入的状态（通常有 Outlined 边框高亮）。
* **Disabled（禁用）:** 不可点击或操作的状态（置灰，变淡，鼠标变为禁用样式）。

### 2. 按钮设计规范

* **主次分明:** 一个主操作区域**只能有一个主按钮（Primary）**，避免多个高饱和度按钮抢夺用户注意力。
* **危险操作:** 删除、注销等不可逆操作，主按钮使用危险色（Error Red）。
* **文案:** 简明扼要，使用动词或动宾短语（如“提交”、“下载报告”），避免模糊词汇。

### 3. 表单设计规范

* **对齐方式:** 标签（Label）建议采用**顶部对齐**（适合移动端或追求填写速度的表单）或**右对齐**（适合 PC 端，视觉动线短）。
* **必填项标记:** 必填字段前使用红色星号 `*` 标记，且必须在表单开头或末尾有明确说明。
* **校验时机:** 优先采用**失去焦点（Blur）时校验**或**行内实时校验**，不要等到用户点击“提交”后才一股脑报错。

### 4. 弹窗与遮罩规范

* **触发源明确:** 模态弹窗必须由用户主动触发（如点击按钮），系统强弹窗需克制。
* **退出路径清晰:** 提供至少 3 种退出弹窗的方式：点击右上角关闭按钮（X）、点击弹窗外的遮罩层（可选）、按下 `Esc` 键。
* **阻断性:** 对话框（Modal）会打断用户当前操作，因此文案必须包含明确的后果提示（如“确定删除此文件吗？此操作不可逆”）。

## 四、 落地与协作建议

1. **建立设计资产库:** 推荐使用 Figma、Sketch 或 Adobe XD 将上述组件打包为 `Components`，利用 `Variants`（变体）管理不同状态。
2. **对接前端代码库:** 尽量对齐成熟的开源 UI 库命名（如 Ant Design、Element Plus 或 Tailwind UI），这能让设计令牌（Design Tokens）直接映射到代码的 CSS 变量中，减少开发沟通成本。

## 五、 进阶与高阶组件列表（Advanced Components）

随着业务复杂度的提升，以下这些复合型或高级组件会频繁出现：

### 1. 复杂数据录入与选择

* **穿梭框 (Transfer):** 用双栏布局展示数据，实现把数据从“左侧源列表”移动到“右侧目标列表”的操作。
* **级联选择器 (Cascader):** 用于多层级数据的选择（如：省市区三级联动、公司组织架构）。
* **滑块 (Slider):** 通过滑动轴在区间内选择一个值或一个范围（如音量、价格区间）。
* **富文本编辑器 (Rich Text Editor):** 集成加粗、加链接、插图等功能的复杂文本域。

### 2. 高级数据展示

* **树形控件 (Tree):** 结构化展示具有层级关系的数据（如文件目录、权限树）。
* **时间轴 (Timeline):** 按时间顺序垂直排列展示事件流（如：物流追踪、审批日志、版本更新记录）。
* **指标卡 (Statistic):** 突出显示核心数据、百分比及同比/环比趋势，常用于仪表盘（Dashboard）。

### 3. 引导与反馈补充

* **步骤条 (Steps):** 引导用户按照既定流程一步步完成任务（如：注册流程、分步表单）。
* **骨架屏 (Skeleton):** 在页面真实数据未加载完成前，先展示出的灰色占位区块，用以缓解用户等待焦虑。
* **空状态 (Empty):** 当搜索无结果、列表无数据或网络错误时展示的提示页面，**必须带有明确的行动引导（如“去创建”按钮）**。

## 六、 进阶设计规范（高级体验与无障碍）

### 1. 无障碍设计规范 (Accessibility / WCAG)

为了保证视障群体或特殊环境下的用户也能正常使用产品，规范应包含：

* **色彩对比度:** 正文字体与背景色的对比度至少达到 **4.5:1**（WCAG AA级标准）；大文本或图标至少达到 **3:1**。
* **焦点可见性 (Focus Ring):** 键盘用户使用 `Tab` 键切换时，当前选中的组件必须有非常明显的聚焦高亮外框，**严禁隐藏 `outline**`。
* **纯色暗示失效:** 区分状态不能仅仅依赖颜色。例如：表单报错除了边框变红，还必须配合**报错图标**和**文字提示**。

### 2. 微动效规范 (Motion)

动效不只是为了好看，更是为了提供空间叙事和状态反馈。

* **时长 (Duration):**
* 出现/展开等显现动效：**200ms - 300ms**（符合人类视觉捕捉习惯）。
* 消失/关闭等隐藏动效：**150ms - 200ms**（需要更快，给用户干净利落的感觉）。


* **缓动函数 (Easing):**
* 严禁使用匀速（Linear）动效，应使用正弦/三次贝塞尔曲线（如 `ease-out` 或 `cubic-bezier(0.4, 0, 0.2, 1)`），模拟物理世界的惯性。



### 3. 全局加载（Loading）规范

* **局部加载:** 如果是组件内部数据刷新（如表格翻页），只在组件内部使用局部 Spin/Loading，不锁死全屏。
* **全屏加载:** 涉及系统级跳转或初始化时，使用带有遮罩的全屏 Loading 或骨架屏，并设置延迟触发（如超过 300ms 未加载完才显示），避免短时间闪烁。

## 七、 页面级标准模板（Templates）

组件规范的最终形态是组合成“页面模板”。规范中应定义以下常见页面的标准解法：

| 页面类型 | 核心组件组合 | 设计要点 |
| --- | --- | --- |
| **列表页 (List)** | 面包屑 + 筛选表单 + 按钮组 + 表格 + 分页器 | 筛选条件超过 4 个需支持折叠；表格需定义自适应宽度和列的对齐方式（文本左对齐，数字右对齐）。 |
| **详情页 (Detail)** | 页面头 + 状态标签 + 描述列表 + 标签页 + 关系图表 | 核心状态（如：审批中、已拒绝）需在页面最显眼处用强视觉标记表现。 |
| **工作台 (Dashboard)** | 栅格布局 + 指标卡 + 复杂图表 (ECharts) | 数据从左到右、从上到下按重要程度递减排列；注意多图表同时加载的性能和响应式适配。 |

## 八、 规范的维护与演进流程

一份好的设计规范不是静态的，而是**动态演进的生命体**：

1. **规范评审会:** 当业务中出现现有组件无法满足的场景时，由设计师与前端代表共同评估：该场景是*特异性需求*（单独定制）还是*通用需求*（沉淀为新组件/新样式）。
2. **版本控制 (Version Control):** 像代码一样管理设计规范。如 `V1.1.0`（新增骨架屏组件，优化了按钮的 Hover 颜色）。
3. **单源真理 (Single Source of Truth):** 确保 Figma 里的设计资产库（Figma Library）与前端的 NPM 组件库版本严格同步，任何一方的改动都必须同步推送到另一方。
